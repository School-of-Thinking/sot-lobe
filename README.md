# sot-lobe

Universal Scales delivery engine for `sotlobe.ai` ‚Äî an Edge-runtime API that serves the SOT (School of Thinking) thinking-tool manifest as raw Markdown to subscriber AI agents.

## Architecture

```
   AI Agent
      ‚îÇ

      ‚ñº
   Vercel Edge ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚ñ∫ Vercel Edge Config (lookupAccessCode)
      ‚îÇ                       ‚îÇ
      ‚îÇ              valid? ‚îÄ‚îÄ‚îò
      ‚ñº
   text/markdown
   (the SOT Universal Scales skill manifest, ~4KB)
      ‚îÇ
      ‚îî‚îÄ‚îÄ‚ñ∫ Supabase async log (sot_access_logs) via next/server `after()`
```

- **Runtime:** Vercel Edge (sub-50ms global)
- **Validation:** Vercel Edge Config (globally replicated, sub-ms reads, free with Vercel)
- **Logging:** Supabase, fire-and-forget via `after()` from `next/server`
- **Content:** Inlined from `content/scales-skill.md` (~4KB, hand-curated procedural skill), regenerated by `scripts/build-manifest.mjs` on every `predev`/`prebuild`

### Why Edge Config and not Upstash?
The subscription dataset is small (‚â§ a few hundred entries) and changes infrequently. Edge Config is purpose-built for this: globally replicated, sub-ms reads, free with Vercel, no extra service to manage. Upstash Redis would be the right call at higher write frequencies or larger datasets.

## Endpoint

```
GET /:access_code
```

| Status | When |
|---|---|
| `200 text/markdown` | `access_code` matches `^sot_[A-Za-z0-9_-]{1,128}$` AND a matching item exists in Edge Config |
| `404 text/plain` | Code missing, malformed, or key not found (no info leak for invalid codes) |
| `503` | Edge Config lookup failure |

Headers on a 200:
- `Content-Type: text/markdown; charset=utf-8`
- `Cache-Control: public, s-maxage=3600, stale-while-revalidate=86400`
- `X-Manifest-Version: <ISO timestamp>` (build time of the inlined manifest)

## Content

The manifest is **`content/scales-skill.md`** ‚Äî a hand-curated procedural skill (currently `sot-scales-universal` v1.2) covering the GBB, DVR, CPV, and TPF cognitive execution protocols.

To update:
1. Edit `content/scales-skill.md` directly (preserve the YAML frontmatter at the top).
2. `npm run build:manifest` regenerates `src/lib/manifest.generated.ts` (also runs automatically on `npm run dev` and `npm run build`).
3. Commit + push. Vercel rebuilds and the new content is live within a minute.

`X-Manifest-Version` on the response is the source file's `mtime`, so subscribers can detect updates without parsing the body.

## Schema (Supabase)

See `supabase/migrations/001_create_sot_lobe_tables.sql`:
- `sot_subscriptions` ‚Äî client name, `access_key` (UNIQUE, must match `^sot_‚Ä¶$`), plan_type, expiry, is_active
- `sot_access_logs` ‚Äî access_code, optional key_id FK, agent_info (User-Agent), accessed_at

RLS is enabled on both. The Edge route uses the service_role key, which bypasses RLS. No anon/authenticated policies are defined yet.

## Deployment

Vercel auto-deploys on push to `main`. Production domain: `https://sotlobe.ai`.

Required environment variables (set via Vercel dashboard or `vercel env`):

| Variable | Used in |
|---|---|
| `EDGE_CONFIG` | Auto-injected when Edge Config is connected to the Vercel project |
| `NEXT_PUBLIC_SUPABASE_URL` | `after()` access logger |
| `SUPABASE_SERVICE_ROLE_KEY` | `after()` access logger |
