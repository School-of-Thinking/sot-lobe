# sot-lobe

Universal Scales delivery engine for `sotlobe.ai` — an Edge-runtime API that serves the SOT (School of Thinking) thinking-tool manifest as raw Markdown to subscriber AI agents.

## Architecture

```
   AI Agent
      │
      │  GET sotlobe.ai/sot_xxxxxxxxx
      ▼
   Vercel Edge ─────► Vercel Edge Config (lookupAccessCode)
      │                       │
      │              valid? ──┘
      ▼
   text/markdown
   (the SOT Skill manifest, ~86KB)
      │
      └──► Supabase async log (sot_access_logs) via next/server `after()`
```

- **Runtime:** Vercel Edge (sub-50ms global)
- **Validation:** Vercel Edge Config (globally replicated, sub-ms reads, free with Vercel)
- **Logging:** Supabase, fire-and-forget via `after()` from `next/server`
- **Content:** Inlined from `content/sot-knowledge/*.md` (~86KB), regenerated by `scripts/build-manifest.mjs` on every `predev`/`prebuild`

### Why Edge Config and not Upstash?
The subscription dataset is small (≤ a few hundred entries) and changes infrequently. Edge Config is purpose-built for this: globally replicated, sub-ms reads, free with Vercel, no extra service to manage. Upstash Redis would be the right call at higher write frequencies or larger datasets.

## Endpoint

```
GET /:access_code
```

| Status | When |
|---|---|
| `200 text/markdown` | `access_code` matches `^sot_[A-Za-z0-9_-]{1,128}$` AND `sot_key:{access_code}` exists in Upstash |
| `404 text/plain` | Code missing, malformed, or key not found (no info leak for invalid codes) |
| `503` | Upstream Redis failure |

Headers on a 200:
- `Content-Type: text/markdown; charset=utf-8`
- `Cache-Control: public, s-maxage=3600, stale-while-revalidate=86400`
- `X-Manifest-Version: <ISO timestamp>` (build time of the inlined manifest)

## Local development

```bash
npm install
cp .env.example .env.local   # fill in real values
npm run dev                  # http://localhost:3000
```

Test the route:
```bash
# 1) Seed a test key in Edge Config (via Vercel dashboard or `vercel env`)
#    Add item: key=`sot_devtest_abc123` value="1" (or a real subscription UUID)
curl http://localhost:3000/sot_devtest_abc123     # 200 + manifest
curl http://localhost:3000/nope                    # 404
curl -I http://localhost:3000/sot_devtest_abc123   # check headers
```

## Content sync

Source of truth: Edward bot's daily-baked ZIP at
`/data/.openclaw/workspace/skills/dist/sot-knowledge-latest.skill` on the VPS.

The local `content/sot-knowledge/` mirrors that bundle's contents. To pull a fresh copy:

```bash
npm run sync:content   # scp .skill, unzip, regenerate manifest
git diff content/      # review what changed before committing
```

Manual rebuild without sync:
```bash
npm run build:manifest
```

## Schema (Supabase)

See `supabase/migrations/001_create_sot_lobe_tables.sql`:
- `sot_subscriptions` — client name, `access_key` (UNIQUE, must match `^sot_…$`), plan_type, expiry, is_active
- `sot_access_logs` — access_code, optional key_id FK, agent_info (User-Agent), accessed_at

RLS is enabled on both. The Edge route uses the service_role key, which bypasses RLS. No anon/authenticated policies are defined yet.

## Deployment

Vercel auto-deploys on push to `main`. Production domain: `https://sotlobe.ai`.

Required environment variables (set via Vercel dashboard or `vercel env`):

| Variable | Used in |
|---|---|
| `EDGE_CONFIG` | Auto-injected when Edge Config is connected to the Vercel project |
| `NEXT_PUBLIC_SUPABASE_URL` | `after()` access logger |
| `SUPABASE_SERVICE_ROLE_KEY` | `after()` access logger |
